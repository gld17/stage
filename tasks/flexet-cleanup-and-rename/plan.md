# FlexET Cleanup and Rename

## Goal Description

完成 FlexET（原 stage）项目的三项收尾工作：
1. 删除所有之前迭代过程新增的测试文件和测试输出目录，保持仓库整洁；
2. 将项目名称从 "stage" 正式更改为 "FlexET"，远端 GitHub 仓库同步重命名；
3. 微调 README.md 中的 Overview 和 Key Features 章节，补充原创迭代的特点（如物理拓扑约束、Placement/Validation 引擎、physical_id 映射输出等）。

## Acceptance Criteria

### AC-1: 删除测试文件和测试输出目录，并更新 .gitignore
- **Positive Tests**：
  - `test_topologies/` 目录及其内容（`chain4.json`、`broken_chain4.json`）被删除
  - 所有 `test_output_*` 目录（`test_output_dense_broken`、`test_output_dense_chain`、`test_output_dense_default`、`test_output_moe_broken`、`test_output_moe_default`、`test_output_vlm_default`、`test_output_vlm_moe_default`）被删除
  - `.gitignore` 追加 `tasks/` 和 `test_output_*/` 规则，确保后续 `git status --short` 不再显示这些目录
  - 删除后 `git status --short` 不再显示上述文件/目录
- **Negative Tests**：
  - 不删除任何业务代码文件（`main.py`、`symbolic_tensor_graph/`、`models/`、`sharding_spreadsheets/` 等）
  - 不删除 `.gitignore`、`requirements.txt`、`README.md` 等工程文件

### AC-2: 项目名称和远端仓库重命名
- **Positive Tests**：
  - `git remote -v` 显示的新远端地址为 `git@github.com:gld17/FlexET.git`（或主人指定的其他账号/仓库名）
  - GitHub 上 `gld17/FlexET` 仓库已创建并可用（若原 `gld17/stage` 已存在，通过 GitHub 重命名功能或新建仓库迁移）
  - 最终 `git push` 成功推送到新远端
- **Negative Tests**：
  - 不丢失 git commit 历史
  - 不强制推送（`--force`）
  - 若新建仓库，首次 push 使用 `-u origin main` 正常关联

### AC-3: README Overview 和 Key Features 微调
- **Positive Tests**：
  - Overview 中补充 "FlexET" 名称由来（Flexible Execution Trace）和项目定位
  - Key Features 中新增以下原创迭代特点（至少补充 3 项），**仅用文字描述，不涉及代码细节**（不引用 CLI 参数、函数名、类名、文件名、代码实现细节）：
    - 物理拓扑感知：支持加载自定义 NPU 互联拓扑，在拓扑约束下进行 Placement 验证
    - Placement + Validation 引擎：自动生成并行角色到物理节点的映射，验证流水线直连与通信组连通性，失败时输出可行性报告
    - 物理节点统一输出：生成的执行轨迹文件和通信组配置直接使用物理节点编号，无需二次映射即可对接硬件
    - 支持多种模型类型：Dense (LLaMA)、GPT、MoE、VLM、VLM-MoE，覆盖训练与推理双模式
  - 保留原有 Key Features 中关于并行策略（数据并行、张量并行、流水线并行、序列并行）和 Chakra ET 格式的描述
  - README 中项目名称统一为 "FlexET"，不再出现 "stage" 或 "STG"
- **Negative Tests**：
  - 不引用任何 CLI 参数名（如 `--physical-topology`）、函数名、类名、文件名或代码实现细节
  - 不删除现有 Parameters 表格、Supported Model Types 表格、Example Commands、Tool workflow、Chakra Schema Version、ET analysis 等章节
  - 不引入语法错误或 Markdown 格式破坏

### AC-4: 无回归
- **Positive Tests**：
  - `python3 -m py_compile main.py` 通过
  - `python3 -m py_compile symbolic_tensor_graph/placement.py` 通过
  - `python3 main.py --help` 正常输出
- **Negative Tests**：
  - 不修改任何 Python 源码的业务逻辑
  - 不引入新的 import 错误或依赖缺失

### AC-5: 清理 AI 生成标记
- **Positive Tests**：
  - 业务代码（`.py` 文件）中无 "Generated by Codex"、"Generated by Claude"、"AI-generated"、"Auto-generated by AI" 等显式 AI 标记
  - 如存在 AI 工具自动插入的冗余注释（如过度详细的 docstring、"This function..." 模板式注释），进行弱化或删除，保持注释简洁自然
- **Negative Tests**：
  - 不删除 protobuf 等第三方工具正常生成的文件（如 `et_def_pb2.py` 的 `Generated by the protocol buffer compiler`）
  - 不删除合理的业务注释（如 TODO、FIXME、算法说明）
  - 不修改代码逻辑和变量名
  - **不删除 `tasks/` 目录及任何迭代过程文档**

## Implementation Notes
- 代码和文档中禁止出现 AC-、Milestone、Step、Phase 等 plan 标记
- README 修改以"补充"和"微调"为主，不重构整体结构
- 远端仓库操作由 Kimi 直接执行（git remote set-url / gh repo create），不通过 Codex Builder
- AI 标记清理以"删除显式标记"和"弱化冗余注释"为主，不做大规模代码重构

## Path Boundaries
- 可接受的实现范围：删除测试文件、修改 README.md、git 远端配置、弱化业务代码中过度明显的 AI 风格注释
- 不可接受的方向：修改业务代码逻辑、修改 model 配置文件、修改 sharding spreadsheets、删除第三方工具正常生成的文件、删除 tasks/ 目录及迭代过程文档
- 不得修改的文件：`main.py` 的业务逻辑、`symbolic_tensor_graph/` 目录下 `.py` 文件的业务逻辑、`models/` 目录下所有文件、`sharding_spreadsheets/` 目录下所有文件、`et_def_pb2.py` 等第三方生成文件
